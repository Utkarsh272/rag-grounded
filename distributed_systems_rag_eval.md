# Distributed Systems: Foundations and Patterns

## 1. Consistency Models

Consistency models define the rules for how and when updates made by one process become visible to other processes in a distributed system. Choosing the right consistency model is one of the most consequential architectural decisions a distributed systems engineer makes, because it directly determines what guarantees the system can offer to application developers and end users.

**Strong consistency** (also called linearizability) guarantees that every read reflects the most recent write, and that all operations appear to execute atomically at a single point in time. A system is linearizable if, for any execution, there exists a valid sequential ordering of operations that is consistent with the real-time ordering of those operations. Linearizability is the gold standard for correctness but comes at a significant cost: it requires coordination between replicas on every write, which increases latency and reduces availability under network partitions. Google Spanner achieves external consistency (a form of linearizability) across globally distributed datacenters using TrueTime, a globally synchronized clock with bounded uncertainty intervals.

**Sequential consistency** is a weaker model where all processes agree on a global ordering of operations, but that ordering does not need to respect real-time order. It is sufficient for many concurrent programming scenarios (it is the model guaranteed by most memory models in languages like Java and C++) but is still expensive to implement in distributed systems because it requires global coordination.

**Causal consistency** weakens the guarantee further: only causally related operations are required to be seen in the same order by all processes. If process A writes X and then writes Y after reading X, any process that sees Y must also see the write to X. However, concurrent operations (those with no causal relationship) may be seen in different orders by different processes. Causal consistency can be implemented without global coordination, making it significantly cheaper than linearizability. Systems like MongoDB (with causal sessions) and Cassandra (with lightweight transactions disabled) operate in this regime.

**Eventual consistency** provides the weakest guarantee: if no new updates are made to a data item, eventually all replicas will converge to the same value. There is no bound on how long convergence takes, and during the convergence window, different replicas may return different values. Despite this weak guarantee, eventual consistency is widely used in high-scale systems — Amazon DynamoDB, Apache Cassandra, and Amazon S3 all default to eventual consistency. The reason is practical: eventual consistency allows systems to remain available and low-latency even under network partitions, sacrificing momentary correctness for durability and availability.

The CAP theorem, formalized by Eric Brewer and proved by Gilbert and Lynch, states that a distributed system can provide at most two of three properties: Consistency (every read receives the most recent write), Availability (every request receives a non-error response), and Partition tolerance (the system continues to operate despite network partitions). Since network partitions are inevitable in real distributed systems, the practical choice is between CP systems (consistent but potentially unavailable during partitions) and AP systems (available but potentially inconsistent). HBase and Zookeeper are CP systems. Cassandra and CouchDB are AP systems. Most modern systems allow this tradeoff to be tuned per operation.

The PACELC theorem extends CAP by noting that even in the absence of partitions, there is always a latency-consistency tradeoff: a system that avoids partitions still must choose between lower latency (by not waiting for all replicas to acknowledge) and stronger consistency (by waiting). This captures the everyday engineering reality more completely than CAP.

## 2. Replication Strategies

Replication — maintaining copies of data on multiple nodes — is the primary mechanism by which distributed systems achieve fault tolerance and high availability. The key design dimensions are: who can accept writes (leader-based vs. leaderless), how writes are propagated to replicas (synchronous vs. asynchronous), and how conflicts are resolved when concurrent writes occur.

**Single-leader replication** (also called master-slave or primary-backup) designates one replica as the leader. All writes go through the leader, which applies them to its local storage and propagates them to followers as a replication log. Reads can be served from followers (offering read scalability) or from the leader only (offering read-your-writes consistency). Followers consume the replication log and apply changes in the same order as the leader. This is the model used by PostgreSQL streaming replication, MySQL binlog replication, and Kafka (where the partition leader is the single writer). The main limitation is that the leader is a write bottleneck and a single point of failure — leader failure requires a failover process that must detect the failure, elect a new leader, and ensure the old leader cannot re-join as leader (to prevent split-brain).

**Multi-leader replication** allows multiple nodes to accept writes. This is essential for multi-datacenter deployments where a single leader would impose cross-datacenter latency on every write. Each datacenter has a local leader, and leaders asynchronously replicate to each other. The fundamental challenge is write conflicts: if two leaders concurrently update the same record, there is no automatic ordering of which write should win. Conflict resolution strategies include last-write-wins (using timestamps, which can be unreliable due to clock skew), application-level conflict resolution (the application receives both conflicting versions and merges them), and CRDTs (data structures that are algebraically designed to merge without conflicts). CouchDB, Cassandra (with multi-datacenter replication), and Google Docs (for collaborative editing) use variants of multi-leader or leaderless replication.

**Leaderless replication** (the Dynamo model) allows any replica to accept writes. The client (or a coordinator node) sends writes to multiple replicas in parallel and considers the write successful when W replicas acknowledge it. Reads are sent to R replicas in parallel, and the client takes the most recent value (determined by version vectors or timestamps). The system is available as long as W + R > N (where N is the total replica count), which provides a quorum-based consistency guarantee. If W + R ≤ N, the system may return stale reads. Amazon DynamoDB, Apache Cassandra, and Riak use leaderless replication. The main advantage is that no failover is needed — any replica can serve reads and writes. The main challenge is handling read repair (correcting stale replicas when a read discovers a discrepancy) and anti-entropy (background reconciliation to ensure all replicas eventually converge).

**Synchronous vs. asynchronous replication** is an orthogonal dimension. In synchronous replication, the leader waits for followers to confirm the write before acknowledging success to the client. This guarantees no data loss on leader failure (the follower has the data) but increases write latency. In asynchronous replication, the leader acknowledges immediately after writing locally, and propagates to followers in the background. This minimizes write latency but means that if the leader fails before replication completes, acknowledged writes may be lost. Semi-synchronous replication (used by MySQL and PostgreSQL) keeps one synchronous follower and the rest asynchronous, providing a balance: no data loss on single-node failure while keeping latency low.

## 3. Consensus Algorithms

Consensus — getting a group of nodes to agree on a single value even in the presence of failures — is the core problem in distributed systems. It underlies leader election, distributed transactions, replicated state machines, and coordination services. The fundamental impossibility result (FLP, Fischer-Lynch-Paterson 1985) proves that no deterministic consensus algorithm can guarantee termination in an asynchronous system where even one process can fail. In practice, systems avoid this by using timeouts to detect failures (making assumptions about timing) or by using randomization.

**Paxos**, proposed by Leslie Lamport in 1989 and published in 1998, is the foundational consensus algorithm. It operates in two phases. In Phase 1 (Prepare/Promise), a proposer sends a prepare message with a proposal number n to a quorum of acceptors. Each acceptor promises not to accept proposals numbered less than n and returns the highest-numbered proposal it has already accepted (if any). In Phase 2 (Accept/Commit), if the proposer receives promises from a quorum, it sends an accept message with value v (the value from the highest-numbered previously accepted proposal, or its own value if none) and number n. Acceptors accept if they have not promised to ignore requests with numbers less than n. Once a quorum of acceptors accept, the value is chosen. Paxos guarantees safety (only one value is ever chosen) but does not guarantee liveness (a proposer can be repeatedly preempted by another proposer). Multi-Paxos extends basic Paxos to handle a sequence of values (a log) by electing a stable leader that can skip Phase 1 for subsequent proposals.

**Raft**, designed by Diego Ongaro and John Ousterhout and published in 2014, was explicitly designed to be more understandable than Paxos while providing equivalent safety guarantees. Raft decomposes consensus into three subproblems: leader election, log replication, and safety. Raft uses a term-based leader election: servers start as followers and become candidates when they time out waiting for a heartbeat from the leader. A candidate requests votes from all servers; it wins the election if it receives votes from a majority. The leader then replicates log entries to followers. A log entry is committed once the leader has replicated it to a majority of servers. Raft's key invariant is the Leader Completeness Property: if a log entry is committed in a given term, it will be present in the logs of all leaders for all higher-numbered terms. This ensures committed entries are never lost. Raft is used in etcd (Kubernetes' coordination store), CockroachDB, TiKV, and Consul.

**Byzantine fault tolerance** (BFT) addresses a harder problem: consensus in the presence of nodes that may behave arbitrarily (send incorrect or conflicting messages, collude, lie). The classic algorithm is PBFT (Practical Byzantine Fault Tolerance), which requires 3f+1 nodes to tolerate f Byzantine faults. BFT algorithms are significantly more expensive (O(n²) message complexity) than crash fault tolerant algorithms like Paxos and Raft. They are used primarily in blockchain systems (where nodes are untrusted) and safety-critical systems. Modern BFT variants include Tendermint (used in Cosmos blockchain) and HotStuff (used in Facebook's Diem/Libra).

**Zookeeper Atomic Broadcast (ZAB)** is the consensus protocol underlying Apache ZooKeeper. Like Raft, it is a leader-based protocol for replicating a sequence of transactions. ZAB has two modes: recovery (electing a leader and synchronizing state after a failure) and broadcast (normal operation where the leader proposes transactions and followers acknowledge them). ZAB guarantees that if a transaction is delivered, all previously proposed transactions are delivered first (prefix property), making it suitable for building coordination primitives like distributed locks, leader election, and configuration management.

## 4. Distributed Transactions and Concurrency Control

Distributed transactions allow a set of operations across multiple nodes or services to execute atomically — either all succeed or all fail — despite node failures and network partitions. They are among the hardest problems in distributed systems because they must combine atomicity, consistency, isolation, and durability (ACID) with the challenges of partial failure and coordination overhead.

**Two-Phase Commit (2PC)** is the standard protocol for atomic commitment across distributed participants. In Phase 1 (Prepare), the coordinator sends a prepare message to all participants. Each participant writes the transaction data to a durable write-ahead log and votes yes (ready to commit) or no (cannot commit, due to a constraint violation or insufficient resources). In Phase 2 (Commit/Abort), if all participants voted yes, the coordinator sends a commit message; if any participant voted no, it sends an abort message. Participants then apply or discard the transaction based on the coordinator's decision. The fundamental problem with 2PC is that it is a blocking protocol: if the coordinator crashes after Phase 1 but before Phase 2, participants are left in an uncertain state (they cannot commit or abort without the coordinator's decision) and must wait indefinitely. This is called the in-doubt window. Systems like PostgreSQL, MySQL (with XA transactions), and Spanner use 2PC internally.

**Three-Phase Commit (3PC)** adds a pre-commit phase between prepare and commit to break the blocking property of 2PC. After receiving yes votes from all participants, the coordinator sends a pre-commit message (essentially "I will commit, get ready") before the final commit. If the coordinator crashes during the pre-commit phase, participants can communicate with each other and determine the outcome: if any participant has received a pre-commit, they can all commit; if none have, they can all abort. 3PC eliminates the in-doubt window but is more expensive (3 round trips instead of 2) and does not handle network partitions correctly (it assumes a synchronous network).

**Saga pattern** is an alternative to distributed transactions that avoids 2PC by breaking a long-running transaction into a sequence of local transactions, each with a compensating transaction that undoes its effects. If a step fails, the saga executes the compensating transactions for all previously completed steps in reverse order. Sagas are eventually consistent (intermediate states are visible) rather than ACID-consistent. They are widely used in microservices architectures (popularized by Chris Richardson) because they do not require a distributed transaction coordinator. The tradeoff is increased application complexity: the developer must define compensating transactions for every step and handle the case where a compensating transaction fails.

**Optimistic Concurrency Control (OCC)** allows transactions to execute without acquiring locks, assuming conflicts are rare. At commit time, the system validates that the data read by the transaction has not been modified by another transaction since the read (using version numbers or timestamps). If a conflict is detected, the transaction is aborted and retried. OCC performs well under low contention (most reads complete without conflict) but degrades under high contention (many transactions abort and retry, wasting work). CockroachDB and Google Spanner use OCC with multi-version concurrency control (MVCC) for read-write conflicts.

**Multi-Version Concurrency Control (MVCC)** maintains multiple versions of each data item, one for each transaction that has modified it. Readers read the version that was current at the start of their transaction (snapshot isolation), without blocking writers. Writers create a new version without overwriting the old one. This eliminates read-write conflicts entirely: reads never block writes and writes never block reads. The tradeoff is storage overhead (multiple versions must be retained until no reader can see them) and the need for a garbage collection process to reclaim old versions. PostgreSQL, MySQL InnoDB, CockroachDB, and Spanner all use MVCC.

## 5. Distributed Storage and Data Partitioning

Data partitioning (sharding) divides a large dataset across multiple nodes, allowing horizontal scaling of both storage and throughput. The two fundamental approaches are range partitioning and hash partitioning, each with distinct tradeoffs.

**Range partitioning** assigns contiguous ranges of the partition key to each node. For example, users A–M go to node 1, N–Z go to node 2. Range partitioning enables efficient range queries (all users starting with "A" can be served from a single node) and sequential scans. The main risk is hotspots: if most writes target a narrow key range (e.g., a time-series workload where all new data has recent timestamps), one node receives all the write traffic while others are idle. HBase and Google Bigtable use range partitioning with automatic tablet splitting when a tablet grows too large.

**Hash partitioning** applies a hash function to the partition key and assigns the result to a node. This distributes data more uniformly but destroys range query locality — a range query must be broadcast to all nodes. Consistent hashing is a variant of hash partitioning that minimizes data movement when nodes are added or removed. In consistent hashing, both nodes and keys are hashed onto a ring. Each key is assigned to the first node clockwise from its position on the ring. When a node is added, it takes over a portion of the next node's key range. When a node is removed, its keys are reassigned to the next node clockwise. Amazon DynamoDB, Apache Cassandra, and Akamai's CDN use consistent hashing. Virtual nodes (vnodes) improve load balancing by assigning each physical node multiple positions on the ring.

**LSM trees (Log-Structured Merge-trees)** are the write-optimized storage engine used by RocksDB, Cassandra, LevelDB, and HBase. All writes go to an in-memory buffer (memtable) and a write-ahead log. When the memtable fills, it is flushed to disk as an immutable sorted file (SSTable). Background compaction merges and sorts SSTables, discarding old versions of keys. LSM trees achieve very high write throughput (sequential disk writes, no random I/O on the write path) at the cost of read amplification (a read may need to check multiple SSTables) and write amplification (a key may be rewritten multiple times during compaction).

**B-trees** are the read-optimized storage engine used by PostgreSQL, MySQL InnoDB, SQLite, and most traditional relational databases. A B-tree maintains a balanced tree of fixed-size pages on disk. Reads require O(log n) page reads (typically 3–4 for large datasets). Writes update pages in-place, requiring a write-ahead log for crash recovery. B-trees provide excellent read performance and support efficient range queries. Their weakness is write amplification under heavy write loads: a single key update may require reading and rewriting a full page (typically 4–16 KB) even if only a few bytes changed.

**Bloom filters** are a probabilistic data structure used in LSM-based storage engines to avoid unnecessary disk reads. Before searching an SSTable for a key, the engine checks the SSTable's Bloom filter. If the filter says the key is absent, the SSTable is skipped entirely. Bloom filters have a tunable false positive rate (they may incorrectly report a key as present) but never produce false negatives (if the filter says absent, the key is definitely absent). This dramatically reduces read amplification in LSM-based systems for lookups of non-existent keys.

## 6. Observability and Distributed Tracing

Observability is the ability to understand the internal state of a distributed system from its external outputs — logs, metrics, and traces. In a microservices or distributed architecture, a single user request may traverse dozens of services; traditional single-process debugging tools are insufficient. The three pillars of observability — metrics, logs, and traces — each serve a distinct purpose and are complementary.

**Metrics** are numerical measurements aggregated over time. They answer questions like "what is the p99 latency of my API endpoint?" and "how many requests per second is my database handling?" Metrics are efficient to collect and store (a single time series rather than per-request data) and are ideal for alerting and dashboarding. The industry-standard metrics system is Prometheus, an open-source monitoring system that scrapes metrics from services via HTTP, stores them in a time-series database, and provides a query language (PromQL) for analysis. Grafana is commonly used as a visualization layer on top of Prometheus. Key metric types include counters (monotonically increasing, e.g., total request count), gauges (current value, e.g., active connections), histograms (distribution of values, e.g., request latency bucketed by range), and summaries (pre-calculated quantiles).

**Structured logs** are machine-readable records of individual events. Unlike unstructured text logs, structured logs are formatted as key-value pairs (typically JSON), making them queryable and aggregatable by log management systems like Elasticsearch, Loki, and Splunk. Each log entry should include a timestamp, severity level, service name, trace ID (to correlate with distributed traces), and the event-specific fields. The `structlog` library (Python) and `slog` (Go 1.21+) are the idiomatic choices for structured logging in their respective ecosystems. Log sampling (recording only a fraction of high-volume events) is essential at scale to control storage costs.

**Distributed tracing** tracks a request as it flows through multiple services, recording the time spent in each service and the relationships between operations (spans). A trace is a directed acyclic graph of spans, where each span represents a single unit of work (an HTTP call, a database query, a cache lookup). Spans are connected by a trace context (a trace ID and parent span ID) propagated through request headers. OpenTelemetry (OTel) is the CNCF standard for instrumentation: a vendor-neutral API and SDK that collects traces, metrics, and logs and exports them to backends like Jaeger (open-source), Zipkin (open-source), Honeycomb, or Datadog. The W3C Trace Context specification standardizes the HTTP headers used to propagate trace context between services (traceparent and tracestate headers).

**The RED method** (Rate, Errors, Duration) is a framework for defining the key metrics every service should expose: the rate of requests per second, the rate of errors per second, and the distribution of request durations. It was popularized by Tom Wilkie at Weaveworks and is widely adopted as a minimum observability baseline. The complementary USE method (Utilization, Saturation, Errors), from Brendan Gregg, applies to infrastructure resources: CPU utilization, queue saturation, and error rates on disks and network interfaces.

**SLOs, SLIs, and SLAs** provide the contractual and operational framework for reliability. A Service Level Indicator (SLI) is a quantitative measure of service behavior, such as availability (fraction of successful requests) or latency (fraction of requests completing within 200ms). A Service Level Objective (SLO) is a target value for an SLI, such as "99.9% of requests complete within 200ms." An SLA (Service Level Agreement) is a contractual commitment to an SLO, with consequences (typically financial credits) for violations. Google's Site Reliability Engineering (SRE) book introduced the concept of error budgets: the allowed amount of downtime or errors within an SLO period. If the error budget is exhausted, new feature launches are paused until reliability is restored.

**Alerting and on-call practices** close the loop between observability and operations. Effective alerts are symptom-based (alerting on user-visible impact, like high error rate or high latency) rather than cause-based (alerting on CPU usage or disk space, which may not affect users). The SRE golden signals — latency, traffic, errors, and saturation — provide a framework for symptom-based alerting. Alert fatigue (too many non-actionable alerts) is a serious operational risk; every alert should have a clear runbook and a clear owner. Tools like PagerDuty and OpsGenie manage on-call rotations and alert routing.
